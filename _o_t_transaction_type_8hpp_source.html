00284     // will successfully verify Alice's signature on any account A instruments, and it will also successfully
00285     // verify Alice's signature on any account B instruments.  This means if you sneakily give the server

00287     // signed instrument, WILL VERIFY SUCCESSFULLY. When the server is loading a transaction for "account A",

00291     // which we do before ever actually using that transaction object. This is why it's important not to
00292     // simply "load up" the transaction and go with whatever IDs are in there, or allow them to overwrite
00293     // the ones we are expecting to be there.

00297     // IDs expected. The # cannot be changed after signing, without invalidating the signature, so we verify the
00298     // signature. This means the transaction item cannot be used on any OTHER transaction except for the one
00299     // it was meant, or by any other acct owned by that same nym, because those IDs are compared whenever each

00312     // expecting it to be when you load the transaction up. You keep them separate, and you compare them to
00313     // make sure they are the same.

00346     // damn well better only read #3, and I will store separately what I expect, versus what I load, and I will compare

00348     // if I expected #3, but actually loaded #5, I'm not going to then take #5 and SAVE IT AS A THREE, just because I
00349     // am in the habit of forcing myself to ONLY SAVE AS A THREE. That could enable someone to "slip" me a 5, and get
00350     // me to re-sign it as a 3!! As long as the signature matches, that part verifies successfully. If I just loaded
00351     // and verified the signature, and forced the actual # to be the one I'm expecting, the a "5" could be used as a "3"
00352     // as long as they had the same owner. I DO want to force the number, since it should be a specific one, but I don't
00353     // want that to ever cause me to get tricked into turning "auto-fixing" an instrument from a different # by substituting
00354     // the correct # onto it just prior to signing, because someone had tricked me into loading it at the wrong time,
00355     // by swapping it for the contract that was meant to be there, etc. That's why I have to compare IDs.
00356     //
00357     // The "Real" server ID is set when the item is first generated. SINCE it was first generated, we go ahead at that one
00358     // time and set the Purported Server ID to match. Ever thereafter, they are separated in a way, with the "real" ID being
00359     // only set as an "expected to load number" and the "purported" ID being "the one we actually loaded" number.
00360     //
00361     // -- We only save to storage the purported number. This prevents you from accidentally saving any number other than
00362     // the one you loaded, into any given instrument, whether it's the number you expected or not. No one can ever give
00363     // you one instrument and "trick" you into signing it with your "expected" IDs substituted. If you slip me a 69, then
00364     // I will also save it as a 69, not as a 5 just because I thought it would be a 5.
00365     // -- We only load from storage to the purported number, not to the real one. This prevents the software from actually
00366     // using the number for anything internally.
00367     // We never load to the real number. (This prevents anyone from "fucking with" your actual ID by something
00368     // they put into a file, since nothing in the datafile will ever be used as the "real" number.)
00369     // We never save the real number. (This insures that we will only save the same number we loaded.)
00370     //
00371     // -- If you slip me a #69 when I expected a #5, you can never trick me into using "#69" on any of my stuff by doing so,
