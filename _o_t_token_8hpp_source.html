00151 /*
00152 
00153  Here's a rough sketch of the protocol:
00154 
00155  Client requests Mint for withdrawal of 100 ithica work hours.
00156 
00157 1) Client blinds and sends N tokens to the server, each worth 100 hours. Client retains the keys.
00158 2) Server responds with a single index, the one the server has chosen for signing.
00159 3) Client replies with 99 keys.
00160 4) Server unblinds 99 tokens (or some randomly-chosen % of those) and verifies them.
00161    He signs the last one and returns it.
00162 5) Client receives signed token, unblinds it, stores it for later.
00163 6) When token is redeemed, it has already been unblinded. So Server simply verifies it.
00164 
00165  LAST NAGGING QUESTION:  Should the server sign the other 99 tokens before unblinding them and verifying?
00166                         In fact, what is it verifying at all?? Certainly not the amount, which is not even in
00167                         the Lucre token. If all it does is verify its signature, then why sign it just to
00168                         verify it?  Why exactly am I sending 99 tokens? What is the server unblinding them
00169                         to look for??  Just to make sure all the IDs are random?  That they aren't spent
00170                         already?
00171                         I think that's it.  The client has assurance he chose his own random IDs, the server
00172                         verifies they are random and not spent already, and the ID portion is the only part
00173                         that has to be randomized.
00174 
00175  UPDATE:
00176  Ben Laurie has confirmed that the Chaumian 99 token requirement does not exist with Lucre. All I have to
00177  do is send a single blinded token. The server signs it and sends it back, and the client unblinds it. Only the
00178  ID itself is blinded -- the server can clearly see the amount and only the Mint key for that denomination will work.
00179  */
