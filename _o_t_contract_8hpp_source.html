00249         // TODO: a contract needs to have certain required fields in order to be accepted for notarization.
00250         // One of those should be a URL where anyone can see a list of the approved e-notary servers, signed
00251         // by the issuer.
00252         //
00253         // Why is this important?
00254         //
00255         // Because when the issuer connects to the e-notary to issue the currency, he must upload the
00256         // asset contract as part of that process. During the same process, the e-notary connects to that
00257         // standard URL and downloads a RECORD, signed by the ISSUER, showing the e-notary on the accepted
00258         // list of transaction providers.
00259         //
00260         // Now the e-notary can make THAT record available to its clients (most likely demanded by their
00261         // wallet software) as proof that the issuer has, in fact, issued digital assets on the e-notary
00262         // server in question. This provides proof that the issuer is, in fact, legally on the line for
00263         // whatever assets they have actually issued through that e-notary. The issuer can make the total
00264         // outstanding units available publicly, which wallets can cross-reference with the public records
00265         // on the transaction servers. (The figures concerning total issued currency should match.)
00266         //
00267         // Of course, the transaction server could still lie, and publish a falsified number instead of
00268         // the actual total issued currency for a given digital asset. Only systems can prevent that,
00269         // based around separation of powers. People will be more likely to trust the transaction provider
00270         // who has good accounting and code audit processes, with code fingerprints, multiple passwords
00271         // across neutral and bonded 3rd parties, insured, etc.  Ultimately these practices will be
00272         // governed by the cost of insurance.
00273         //
00274         // But there WILL be winners who arise because they implement systems that provide trust.
00275         // And trust is a currency.
00276         //
00277         // (Currently the code loads the key FROM the contract itself, which won't be possible when
00278         // the issuer and transaction provider are two separate entities. So this sort of protocol
00279         // becomes necessary.)
