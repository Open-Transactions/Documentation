00147 // A nym contains a list of master credentials, via OTCredential.
00148 // The whole purpose of a Nym is to be an identity, which can have
00149 // master credentials.
00150 //
00151 // Each credential is like a master key for the Nym's identity,
00152 // which can issue its own subkeys.
00153 //
00154 // Each subkey has 3 key pairs: encryption, signing, and authentication.
00155 // Not all subcredentials are a subkey. For example, you might have a
00156 // subcredential that uses Google Authenticator, and thus doesn't contain
00157 // any keys, because it uses alternate methods for its own authentication.
00158 //
00159 // Each OTCredential contains a "master" subkey, and a list of subcredentials
00160 // (some of them subkeys) signed by that master.
00161 //
00162 // The same class (subcredential/subkey) is used because there are master
00163 // credentials and subcredentials, so we're using inheritance for "subcredential"
00164 // and "subkey" to encapsulate the credentials, so we don't have to repeat code
00165 // across both.
00166 // We're using a "has-a" model here, since the OTCredential "has a" master
00167 // subkey, and also "has a" list of subcredentials, some of which are subkeys.
00168 //
00169 // Each subcredential must be signed by the subkey that is the master key.
00170 // Each subkey has 3 key pairs: encryption, signing, and authentication.
00171 //
00172 // Each key pair has 2 OTAsymmetricKeys (public and private.)
00173 //
00174 // I'm thinking that the Nym should also have a key pair (for whatever is
00175 // its current key pair, copied from its credentials.)
00176 //
00177 // the master should never be able to do any actions except for sign subkeys.
00178 // the subkeys, meanwhile should only be able to do actions, and not issue
00179 // any new keys.

00192 // THE MASTER CREDENTIAL (below -- OTCredential)
00193 //
00194 // Contains a "master" subkey,
00195 // and a list of subcredentials signed by that master.
00196 // (Some of which are subkeys, since subkey inherits from
00197 // subcredential.)
00198 // Each subcredential can generate its own "credential" contract,
00199 // even the master subcredential, so an OTCredential object
00200 // actually may include many "credentials." (That is, each may be
00201 // issued at separate times. Each may be registered on a server at
00202 // separate times. Etc.)
00203 //
00204 // Each nym has multiple OTCredentials because there may be
00205 // several master keys, each with their own subcredentials.
00206 //
00207 // ------------------------------------------------
00208 // Two things to verify on a master credential:
00209 //
00210 // 1. If you hash m_pstrSourceForNymID, you should get m_pstrNymID.
00211 // 2. m_pstrSourceForNymID should somehow verify m_Masterkey.GetContents().
00212 //    For example, if m_pstrSourceForNymID contains CA DN info, then GetContents
00213 //    should contain a verifiable Cert with that same DN info. Another example,
00214 //    if m_pstrSourceForNymID contains a public key, then m_Masterkey.GetContents
00215 //    should contain that same public key, or a cert that contains it. Another example,
00216 //    if m_pstrSourceForNymID contains a URL, then m_Masterkey.GetContents should contain
00217 //    a public key found at that URL, or a public key that, when hashed, matches one of
00218 //    the hashes posted at that URL.
00219 //
