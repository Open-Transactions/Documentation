00366 // A Nym cannot act as "agent" for another Nym.
00367 // Nor can a Group act as "agent" for a Nym. Why not? Because:
00368 //
00369 // An entity is COMPOSED of its voting groups and its employee Nyms.
00370 // These don't merely act "on behalf" of the entity, but in fact they comprise the entity.
00371 // Therefore the entity can use voting groups and employee Nyms to make decisions BECAUSE
00372 // IT **HAS** voting groups and employee nyms that it can use.
00373 //
00374 // Whereas an individual Nym is NOT composed of voting groups or employee Nyms. So how could
00375 // he designate to act "on his behalf" something that does not even exist?
00376 // So we must ask, how can another Nym, then, be appointed to act as my agent without some
00377 // agreement designating him as such? And that agreement is the entity itself, which is nothing
00378 // more than an agreement between several owners to designate agents to operate according to their
00379 // interests.
00380 
00381 // To directly appoint one Nym to act on behalf of another, yet WITHOUT any agreement in place,
00382 // is to behave as one is the same as the other. But still, who has the key? If both have absolutely
00383 // access and rights to the same key, then why not just both keep a copy of it? In which case now
00384 // it really IS only one Nym in reality, as well as in the software.
00385 // But if one prefers to have his private key, and another his, then they will begin as separate and
00386 // independent individuals. One Nym will not be found within the other as a int64_t-lost, under-developed
00387 // twin!
00388 // Just as reality enforces separate individuals, so does the software end up in the situation where
00389 // two separate Nyms now wish to act with one as agent for the other. This wish is perfectly valid and
00390 // can be accommodated, but logically the software cannot provide the same ONENESS of ACTUALLY SHARING
00391 // THE PRIVATE KEY IN REAL LIFE (which allows the software to ACTUALLY only deal with a single key),
00392 // as opposed to the oneness of "we have separate keys but we want to act in this way". There is oneness,
00393 // and then there is oneness. OT is about contracts between Nyms, and therefore that is the mechanism
00394 // for implementing ANY OTHER FORM OF AGENCY. Whether that agent is your "best man" or a board of voters,
00395 // and whether the agent acts on behalf of you, or some corporation or democracy, either way, either his
00396 // key is the same bits as your key, or they are separate keys in which case there is an agreement
00397 // between them. Otherwise OT simply does not know which agents have authority--and which do not. This
00398 // knowledge is necessary just for being able to function, and is imposed by the natural law.
00399

00401 // but more usefully, common configurations such as new democracy, new corporation, new board-with-veto, etc.
