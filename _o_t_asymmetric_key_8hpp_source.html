00271     // We're moving to a system where the actual key isn't kept loaded in
00272     // memory except under 2 circumstances:   1. We are using it currently,
00273     // and we're going to destroy it when we're done with it.  2. A timer
00274     // is running, and until the 10 minutes are up, the private key is available.
00275     // But: Presumably it's stored in PROTECTED MEMORY, either with specific
00276     // tricks used to prevent swapping and to zero after we're done, and to
00277     // prevent core dumps, or it's stored in ssh-agent or some similar standard
00278     // API (gpg-agent, keychain Mac Keychain, etc) or Windows protected memory
00279     // etc etc. Inside OT I can also give the option to go with our own security
00280     // tricks (listed above...) or to keep the timer length at 0, forcing the
00281     // password to be entered over and over again. IDEA: When the user enters
00282     // the passphrase for a specific Nym, hash it (so your plaintext passphrase isn't
00283     // stored in memory anywhere) and then USE that hash as the passphrase on
00284     // the actual key. (Meaning also that the user will not be able to use his
00285     // passphrase outside of OT, until he EXPORTS the Nym, since he would also have
00286     // to hash the passphrase before manipulating the raw key file.)
00287     // At this point, we have the hash of the user's passphrase, which is what we
00288     // actually use for opening his private key (which is also normally kept in
00289     // an encrypted form, on the hard drive AND in RAM!!) So everything from above
00290     // still applies: I don't want to reveal that hash, so I store it using tricks
00291     // to secure the memory (I have to do this part anyway, ANYTIME I touch certain
00292     // forms of data), or in ssh-agent, and so on, except a timer can be set after
00293     // the user first enters his passphrase. For ultimate security, just set the
00294     // timer to 0 and type your passphrase every single time. But instead let's
00295     // say you set it to 10 minutes. I don't want to store that password hash,
00296     // either. (The hash protects the plaintext password, but the hash IS the ACTUAL
00297     // password, so while the plaintext PW is protected, the actual one is still not.)
00298     // Therefore I will select some random data from OpenSSL for use as a TEMPORARY
00299     // password, a session key, and then encrypt all the hashes to that session key.
00300     // This way, the actual passphrases (hashed version) do NOT appear in memory, AND
00301     // NEITHER DO the plaintext versions! You might ask, well then why not just encrypt
00302     // the plaintext passphrase itself and use that? The answer is, to prevent making
00303     // it recoverable. You don't even want someone to get that session key and then
00304     // recover your PLAINTEXT password! Maybe he'll go use it on a thousand websites!
00305     //
00306     // Next: How to protect the session key (an OTSymmetricKey) from being found?
00307     // First: destroy it often. Make a new session key EVERY time the timeout happens.
00308     // Also: use it in protected memory as before. This could ALWAYS have a timeout
00309     // of 0! If it's in ssh-agent, what more can you do? At least OT will make this
00310     // configurable, and will be pretty damned secure in its own right. Ultimately
00311     // the best solution here is an extern hardware such as a smart card.
